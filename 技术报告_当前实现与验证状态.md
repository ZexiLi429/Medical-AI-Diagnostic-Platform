## 2. 当前总体结论

截至当前状态，项目已经完成了基于 OHIF Viewer 的 AI 分割能力接入，主要包括：

1. 基于 MedSAM 的标准框选分割。
2. 基于 LiteMedSAM 的轻量快速分割。
3. 基于 MedSAM-2 的 3D 全序列追踪分割。

- Orthanc 未运行。

- 因此无法完成依赖 Orthanc 的真实 DICOM 全链路联调。

- 前端已经可以启动。
- AI 后端已经可以启动。

## 3.3 P2：MedSAM-2 3D 全序列追踪

已完成内容：

- 前端新增“3D 全序列追踪 (SAM-2)”功能入口。
- 用户在当前切片使用 Rectangle 工具框选目标后，可触发 3D 追踪。
- 前端会读取：
  - 当前 `SeriesInstanceUID`
  - 当前关键切片索引
  - RectangleROI 对应 bbox
- 后端 MedSAM-2 服务会：
  - 按序列从 Orthanc 下载全部切片
  - 初始化 SAM2 video predictor
  - 从关键切片开始向前后传播
  - 返回每一层分割结果的 RLE mask
- 前端再将每层 mask 写回到 labelmap volume。
- 完成后提示总体积分割体积，并尝试切换到 3D 视图。

实现意义：

- 从单层分割提升到整套序列的连续追踪。
- 满足三维病灶/器官分割与体积分析需求。

## 4. 后端服务架构说明

当前后端已拆分为四个服务：

## 4.1 MedSAM 标准服务（端口 8000）

职责：

- 提供标准 MedSAM 分割能力。
- 支持截图式推理和真实 DICOM 推理。

关键接口：

- `GET /health`
- `POST /segment`
- `POST /points`
- `POST /mask`
- `POST /auto_liver`
- `POST /segment_dicom`

核心逻辑：

1. 接收前端截图或 DICOM 定位信息。
2. 根据 bbox 或点提示生成 prompt。
3. 调用 MedSAM 模型完成分割。
4. 输出 mask。
5. 将 mask 编码为 RLE。
6. 返回叠加图、RLE、置信度、体积等结果。

---

## 4.2 LiteMedSAM 服务（端口 8002）

职责：

- 提供轻量化快速分割。

关键接口：

- `GET /health`
- `POST /segment`
- `POST /points`
- `POST /segment_dicom`

核心逻辑：

1. 将图像缩放到 `256×256`。
2. 使用 TinyViT 编码器提取特征。
3. 使用框选或点提示生成分割结果。
4. 返回 RLE、置信度、体积估计和叠加图。

---

## 4.3 MedSAM-2 3D 服务（端口 8003）

职责：

- 对整个 CT/MR 序列执行三维连续追踪分割。

关键接口：

- `GET /health`
- `POST /session/create`
- `POST /session/{session_id}/segment`
- `DELETE /session/{session_id}`

核心逻辑：

1. 根据 `SeriesInstanceUID` 在 Orthanc 中查找整套序列。
2. 将所有切片下载为帧序列。
3. 初始化 SAM2 predictor。
4. 使用关键切片 bbox 作为初始目标。
5. 向前后切片传播分割。
6. 为每一层输出 RLE mask。
7. 汇总总体积分割体积。

---

## 4.4 LLM 诊断服务（端口 8001）

职责：

- 将结构化医学数据转换为可读诊断报告。

关键接口：

- `GET /health`
- 诊断生成接口（文本/多模态）

核心逻辑：

1. 接收病例和影像分析信息。
2. 拼接标准化放射学 prompt。
3. 调用 GPT 或 Claude。
4. 输出结构化诊断报告。

## 7. 为什么当前机房环境不能完整联调

当前机房电脑的主要限制：

1. `docker` 命令不存在。
2. 当前环境大概率没有管理员权限。
3. Orthanc 默认依赖 Docker Compose 启动。

因此机房环境下无法直接使用项目默认的 Orthanc 启动方案。

这不是项目功能未实现，而是部署环境受限导致的联调中断。
